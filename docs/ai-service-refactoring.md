# Immich OCR 与人脸识别独立服务化改造方案

本文档详细分析了将 Immich 的 OCR 和人脸识别功能改造成独立服务的技术方案。通过标准化 API 接口，可支持商品扫描、智能家居监控等丰富场景。

## 1. 改造目标与收益

将原本作为 Immich 内部组件的 ML 服务解耦，对外提供通用的 AI 能力。

*   **目标场景**:
    *   **商品扫描**: 利用 OCR 识别商品包装文字，或结合 CLIP 进行以图搜图。
    *   **智能家居监控**: 利用人脸识别 API，实现家庭成员识别、陌生人报警。
*   **核心收益**:
    *   **复用性**: 一套服务支持多种业务，无需重复部署。
    *   **灵活性**: 独立扩展，按需调用。

---

## 2. 现状差距分析

Immich ML 服务目前是一个**无状态的计算服务**，与通用 AI 服务存在以下差距：

| 功能模块 | Immich ML 现状 | 通用服务需求 | 差距 |
| :--- | :--- | :--- | :--- |
| **OCR** | 输入图片 -> 输出文本/坐标 | 输入图片 -> 输出文本/结构化数据 | **差距小**。仅需标准化 API 响应格式。 |
| **人脸识别** | 输入图片 -> 输出 **特征向量** | 输入图片 -> 输出 **人员身份 (Who is this?)** | **差距大**。ML 服务不存储人脸库，无法进行身份比对。 |
| **鉴权** | 无鉴权 (依赖 Docker 网络隔离) | API Key / Token 鉴权 | **需新增**。对外服务必须有安全机制。 |

---

## 3. 改造方案：构建 AI Gateway

建议保留原有的 ML 服务（计算层）不变，在其上层构建一个轻量级的 **AI Gateway (API 网关)**，负责业务逻辑封装、鉴权和状态管理。

### 3.1 架构设计

```mermaid
graph TD
    Client[第三方应用/智能家居] -->|HTTP API| Gateway[AI Gateway (New)]
    Gateway -->|计算请求| ML_Service[Immich ML Service (Existing)]
    Gateway -->|数据存储| DB[(Redis/SQLite)]
```

### 3.2 接口定义 (OpenAPI/Swagger)

#### 3.2.1 通用 OCR 接口
*   **POST /v1/ocr/scan**
*   **功能**: 识别图片中的所有文字。
*   **请求**: `image` (file/base64), `language` (optional).
*   **响应**:
    ```json
    {
      "text": "可口可乐\n净含量: 330ml",
      "segments": [
        {"text": "可口可乐", "box": [10, 10, 100, 50], "score": 0.98}
      ]
    }
    ```

#### 3.2.2 人脸识别闭环接口 (重点改造)
为了支持“是谁”的识别，Gateway 需要维护一个轻量级向量库。

*   **POST /v1/face/register** (注册人脸)
    *   **输入**: `image`, `person_name` (e.g., "张三")
    *   **逻辑**: 调用 ML 服务提取向量 -> 存入 DB (Name + Vector)。
*   **POST /v1/face/recognize** (识别人脸)
    *   **输入**: `image`
    *   **逻辑**: 调用 ML 服务提取向量 -> 在 DB 中搜索最相似向量 (Cosine Similarity) -> 返回 `person_name`。
    *   **响应**: `{"person": "张三", "confidence": 0.92}`

### 3.3 技术选型建议

1.  **AI Gateway**: 使用 **FastAPI** (Python) 或 **Go** 开发。
2.  **向量存储**:
    *   **轻量级**: 使用 **Faiss** (内存索引) 或 **SQLite-vss**。
    *   **生产级**: 使用 **Qdrant** 或 **Milvus**。
3.  **鉴权**: 实现简单的 `X-API-Key` 验证。

---

## 4. 实施步骤

1.  **部署 ML 服务**: 保持 `immich-machine-learning` 容器独立运行。
2.  **开发 AI Gateway**:
    *   实现 `/v1/ocr`：透传请求给 ML 服务，格式化响应。
    *   实现 `/v1/face`：引入向量库，实现“注册-识别”逻辑。
3.  **对接应用**:
    *   **商品扫描**: 调用 OCR 接口，通过正则表达式提取商品名/条形码。
    *   **Home Assistant**: 编写插件调用 `/v1/face/recognize`，实现人脸门锁/监控联动。

## 5. 总结

*   **OCR**：改造简单，主要工作是 API 封装。
*   **人脸识别**：需要增加**状态存储层**（向量库），这是实现通用“身份识别”的关键。
*   此方案既利用了 Immich 强大的推理能力，又解耦了业务逻辑，非常适合作为家庭实验室的基础 AI 服务。
