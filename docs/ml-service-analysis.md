# 机器学习 (ML) 服务深度分析

本文档详细分析了 Immich 机器学习服务的架构、业务流程、核心模块技术细节以及扩容策略。

## 1. Server 批量上传与 ML 处理流程

当用户批量上传照片时，Immich Server 不会同步执行机器学习任务，而是采用 **异步、级联且具备容错机制** 的处理流程，以保证高并发下的系统稳定性。

### 1.1 异步任务流

整个流程基于 **BullMQ** 任务队列，分为以下四个阶段：

1.  **元数据提取 (Stage 1)**
    *   照片上传并写入磁盘后，Server 将 `AssetExtractMetadata` 任务压入队列。
2.  **缩略图生成 (Stage 2 - 前置条件)**
    *   元数据提取完成后，触发 `AssetGenerateThumbnails` 任务。
    *   **关键点**：由于 ML 模型（如 CLIP 或人脸识别）通常需要调整大小后的图像（如 224x224），因此缩略图生成是 ML 任务启动的**必要前置条件**。
3.  **ML 任务并发分发 (Stage 3)**
    *   缩略图生成成功后，`JobService` 会一次性将以下任务并行分发到各自的队列：
        *   **SmartSearch (CLIP)**: 生成图像特征向量，用于语义搜索。
        *   **AssetDetectFaces**: 检测照片中的人脸。
        *   **OCR**: 提取照片中的文字。
4.  **二级触发 (Stage 4)**
    *   如果 `AssetDetectFaces` 发现了人脸，会进一步触发 `FacialRecognition` 任务，进行人脸特征提取与聚类。

### 1.2 可靠性与稳定性保障

为了在大规模上传场景下保证 ML 服务不崩溃且任务不丢失，系统实施了多层保障：

*   **Server 侧 (NestJS)**
    *   **任务持久化**: 所有任务存储在 Redis 中，Server 重启后任务不丢失。
    *   **故障转移 (Failover)**: 支持配置多个 ML 服务 URL。当某个节点请求失败（如连接拒绝）时，自动重试下一个节点。
    *   **健康检查**: 定期向 ML 服务发送 `/ping`，自动标记并隔离不健康的节点。

*   **ML 服务侧 (Python FastAPI)**
    *   **并发控制**: 使用 Gunicorn 多进程 + ThreadPoolExecutor 线程池，避免阻塞异步事件循环。
    *   **内存优化**: 实现了 **延迟加载 (Lazy Loading)** 和 **自动卸载 (TTL)** 机制。闲置模型会自动释放内存，防止内存溢出。
    *   **硬件自适应**: 自动识别环境（CPU, CUDA, OpenVINO, ARM NPU）并选择最优推理后端。
    *   **Docker 监控**: 内置 `healthcheck.py`，配合 Docker 引擎实现自动重启。

## 2. ML 服务业务模块与微服务架构

Immich 的 ML 服务目前是一个基于 FastAPI 的单体微服务，包含以下核心模块：

### 2.1 业务模块概览

1.  **CLIP (智能搜索)**
    *   **功能**: 生成图像和文本的特征向量，支持自然语言搜索（如“海边的日落”）。
    *   **组成**: 文本编码器 + 视觉编码器。
2.  **Facial Recognition (人脸识别)**
    *   **功能**: 人脸检测 + 特征提取。
    *   **模型**: RetinaFace (检测) + ArcFace (识别)。
3.  **OCR (文字识别)**
    *   **功能**: 提取图片中的文字信息。
    *   **模型**: PaddleOCR (PP-OCR)。

### 2.2 微服务拆分与扩容分析

**现状**：
目前架构为**单体微服务**。虽然代码逻辑上模块解耦，但 Server 端和 ML 服务端的路由机制不支持按任务类型拆分。

**拆分限制**：
1.  **Server 端路由**: Server 将所有 ML URL 视为同构节点，不支持“将 OCR 任务发给节点 A，将 CLIP 任务发给节点 B”的路由策略。
2.  **ML 服务端开关**: ML 服务没有硬性的“禁用特定模型”开关，收到请求即会尝试加载模型。

**扩容建议**：
*   **水平扩容 (推荐)**: 部署多个完整的 `immich-machine-learning` 容器，在 Server 设置中填入所有 URL。Server 会自动进行负载均衡。
*   **硬件分组**: 虽然不能按业务拆分，但可以部署 GPU 实例和 CPU 实例，利用 ML 服务内部的硬件自适应能力。

## 3. 核心功能技术详解

这两个核心模块均基于 **ONNX Runtime** 运行，旨在提供跨平台的高性能推理。

### 3.1 OCR (光学字符识别)

Immich 的 OCR 功能用于提取图片文本以支持全文搜索。

*   **底层引擎**: **RapidOCR** (基于百度 PaddleOCR 的 ONNX 优化版)。
*   **模型架构**: 两阶段架构。
    *   **文本检测**: 使用 `PP-OCRv4/v5` 检测模型定位文字包围盒 (Bounding Box)。
    *   **文本识别**: 使用 `PP-OCRv4/v5` 识别模型将图像块转换为文字。
*   **处理流程**:
    1.  **预处理**: 图像缩放（长边对齐 960px）、归一化。
    2.  **检测**: 输出文字区域多边形框。
    3.  **矫正**: 对裁剪出的文字区域进行透视变换 (Perspective Transform) 以校正倾斜。
    4.  **识别**: 输出文字内容及置信度。
*   **语言支持**: 支持多语言模型（中、英、韩、日等），但**不支持自动语种检测**，需在设置中指定首选模型。

### 3.2 人脸识别 (Facial Recognition)

用于自动识别人物并进行聚类归档。

*   **底层库**: **InsightFace**。
*   **模型选择**: 默认 **Buffalo_L** (性能均衡)，可选 `Buffalo_S` (轻量) 或 `AntelopeV2` (高精度)。
*   **核心组件**:
    *   **RetinaFace (检测)**: 定位人脸并输出 5 个关键点（眼、鼻、嘴）。
    *   **ArcFace (识别)**: 将人脸转换为 **512 维特征向量 (Embedding)**。
*   **处理流程**:
    1.  **检测**: 输出人脸框和关键点。
    2.  **对齐 (Alignment)**: 基于关键点进行仿射变换，将人脸“扶正”并裁剪为标准尺寸 (112x112)。这是识别精度的关键。
    3.  **特征提取**: 输出 512 维向量。
    4.  **聚类**: *Server 端执行*。使用改进的 DBSCAN 算法，基于向量距离（默认阈值 0.6）将同一人的面部归类。
